# Project Intake

Use this skill when starting a new project so the work begins with the right problem framing, constraints, ownership, and success criteria.

## Purpose

This skill helps convert a vague idea, request, or opportunity into a clear project brief that can guide execution and decision-making.

## Context Check

Before using this skill:

- Identify the current project, domain, or life area
- Identify any existing source-of-truth files, notes, or prior decisions
- Confirm whether this is net-new work, a continuation, or a rescue effort
- Confirm who the stakeholders and audience are
- If key context is missing, ask one targeted question with likely options, brief tradeoffs, and a recommended default, or state assumptions explicitly when reasonable

## When To Use

- A new project is being proposed
- A project exists but is still poorly framed
- Multiple people are operating with different assumptions
- Scope, ownership, or success criteria are unclear

## Core Principles

- Name the objective clearly
- Separate goals from tactics
- Surface constraints early
- Make assumptions visible
- Define success in observable terms
- Clarify what is out of scope

## Workflow

## 1. Define The Objective

Clarify:

- What are we trying to accomplish?
- Why does it matter now?
- What problem is being solved?

## 2. Capture Context

Summarize:

- Relevant background
- Prior attempts or existing assets
- Stakeholders and decision-makers
- Why the project exists now

## 3. Identify Constraints

Document:

- Time constraints
- Budget constraints
- Technical constraints
- Organizational constraints
- Known risks or dependencies

## 4. Define Success Criteria

Write down:

- What success looks like
- What must be true for this to count as a good outcome
- What metrics or milestones matter
- What failure would look like

## 5. Clarify Scope

State:

- What is in scope
- What is out of scope
- What can wait until later

## 6. Recommend A Starting Plan

End with:

- The next best steps
- Suggested owners if relevant
- Immediate open questions
- Any decision that needs to be made before execution starts

## Output Structure

1. Objective
2. Why now
3. Background and context
4. Stakeholders
5. Constraints
6. Success criteria
7. Scope and non-goals
8. Risks and open questions
9. Recommended next steps

## Quality Bar

A good project intake:

- Frames the real problem clearly
- Makes constraints visible
- Defines success concretely
- Reduces ambiguity before execution
- Prevents avoidable rework later

## Common Failure Modes

- Treating a request as a fully defined project
- Skipping constraints
- Starting execution before naming success criteria
- Letting scope expand without intent
