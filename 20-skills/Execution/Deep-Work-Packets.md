# Deep Work Packets

Use this skill to prepare a focused morning deep-work packet that turns one priority into a bounded `90-120 minute` session with the right supporting context.

## Purpose

This skill helps convert an evening planning choice into a concrete morning work container. It is designed for deep reasoning, writing, synthesis, and problem framing, not for reactive work.

## Context Check

Before using this skill:

- Identify the current project and the relevant source-of-truth docs
- Confirm the anchor task, ideally from Linear, but allow a plain-English task when needed
- Confirm the intended deliverable for the next morning
- Check whether the work is right-sized for a `90-120 minute` session
- Check whether the source base is strong enough or whether the packet needs one short constructed doc
- If key context is missing, ask one targeted question with likely options, brief tradeoffs, and a recommended default, or state assumptions explicitly when reasonable

## When To Use

- During a weekday evening shutdown ritual
- When you want one morning work block to produce a concrete artifact
- When the next step requires focused context assembly before starting
- When a Linear issue exists but still needs narrowing into a real session

## Core Principles

- One anchor task per packet
- One concrete deliverable per session
- Protect the `90-120 minute` time box
- Prefer existing project docs before creating new synthesis
- Keep the packet curated, not bloated
- Build for reMarkable and distraction-free use
- Make loop closure back into Linear and project docs explicit

## Workflow

## 1. Choose The Anchor

Start from:

- A Linear issue link or ID, preferred
- A plain-English task description, accepted as a temporary anchor

Clarify the real morning target:

- What should exist by the end of the session
- Whether the task is reasoning, writing, synthesis, or problem framing
- Whether the task is too broad for one session

## 2. Right-Size The Session

Pressure-test the anchor against the flow-state constraints:

- Is the task clear enough to start cleanly
- Is it challenging but not overwhelming
- Can it produce a real artifact in `90-120 minutes`

If not, narrow it into a specific slice before building the packet.

## 3. Define The Deliverable

Write one concrete finish line:

- `By the end of this session, I want to have produced _____`

Prefer rough but real artifacts:

- Memo draft
- Outline
- Decision
- Spec revision
- Analysis write-up
- Research synthesis

## 4. Gather Supporting Material

Propose `2-4` supporting docs maximum.

For each doc, note:

- Why it is included
- What role it should play in the session

Prefer:

- Existing project docs
- Relevant issue context
- Existing research or working notes
- The smallest useful source set

## 5. Decide Whether A Constructed Doc Is Needed

Use existing docs first.

Create one short constructed doc only when the source base is insufficient and the missing context would materially slow or weaken the morning session.

Allowed constructed-doc modes:

- `Decision brief`, default
- `Research memo`, when the blocker is missing background or evidence

If retrieval is weak and the system cannot confidently assemble a good packet, it should say so and ask for `1-2` candidate docs or links rather than bluff.

## 6. Confirm Before Assembly

Run a short confirmation loop:

1. Confirm the deliverable
2. Confirm the selected docs
3. Confirm whether a constructed doc is needed

## 7. Build The Packet

Create one merged PDF briefing book as the main artifact.

Default front section length:

- `2` pages

Allow:

- `3` pages when the issue is genuinely ambiguous or cross-cutting

Treat `4+` front-section pages as a signal that the anchor is too broad or the packet is doing too much.

The packet should use one fixed template:

1. Start ritual page
2. Session goal and target deliverable
3. Short synthesized morning brief
4. Supporting-doc rationales
5. Appended source docs
6. System-update closeout page

## 8. Save The Artifacts

Write outputs into a dated briefing folder containing:

- The merged PDF briefing book
- A small manifest with anchor, deliverable, included docs, rationales, and constructed-doc status
- Any temporary constructed doc used to assemble the packet

## Output Structure

1. Anchor task
2. Session deliverable
3. `90-120 minute` scope check
4. Start ritual
5. Morning brief
6. Supporting docs with one-line rationale each
7. Closeout prompts:
   - What did I produce?
   - What should I update in Linear?
   - What file or doc should I touch next?
   - What is the next obvious step?

## Quality Bar

A good deep-work packet:

- Centers one important piece of work
- Makes the finish line concrete
- Uses a bounded, well-curated doc set
- Helps you start quickly the next morning
- Preserves enough structure to close the loop afterward
- Feels like a work container, not a mini-report

## Common Failure Modes

- The anchor task is too broad for one session
- The deliverable is vague
- Too many supporting docs are included
- A constructed doc is created when existing docs were enough
- The packet hides low-confidence retrieval instead of surfacing it
- The front section becomes the work instead of preparing the work
